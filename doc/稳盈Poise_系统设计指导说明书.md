# 稳盈 / Poise · 系统设计指导说明书

> **产品全称**：资金预测与流动性管理策略智能体（Cash Forecasting & Liquidity Strategy Agent）
> **产品定位**：财资人员的智能参谋——出预测与策略建议，不自动执行交易
> **文档视角**：面向工程实现的系统设计指导（架构 / 模块 / 数据模型 / 算法 / 接口）
> **配套文档**：《稳盈 · 产品需求说明书》（Word，面向业务）
> **版本**：V0.2（产品命名落定）
> **范围**：单体企业 MVP；在数据模型与决策框架上预留集团演进扩展点

---

## 0. 设计总纲

本系统（**稳盈 / Poise**）实现一条闭环：**感知 → 预测 → 推理 → 建议 → 反馈学习**。

核心架构立场（已在需求阶段确认）：

| 决策项 | 取定 | 设计含义 |
|---|---|---|
| 服务对象 | 单体企业（单一法人） | 实体维度先留 `entity_id` 钩子，暂只一个值 |
| 决策权限 | 参谋（人在环中） | 不实现交易执行；可解释性与留痕是硬要求 |
| 核心卖点 | 策略最优 | 决策引擎是皇冠模块，须给精确最优解 |
| 求解架构 | **运筹优化内核 + LLM 编排** | MILP 求最优，LLM 做翻译/编排/解释 |
| 优化目标 | 净收益最大（主）+ 缺口预警（辅） | 安全底线=硬约束，缺口=软预警信号 |
| 时间粒度 | 13 周滚动（按周） | 模型 13 个时段，每周滚动重优化 |

## 1. 系统架构

### 1.1 分层架构

```
┌─────────────────────────────────────────────────────────────┐
│  交互层 (Interaction)                                         │
│  对话编排 · 看板/可视化 · 主动推送 · 方案对比 · what-if 沙盘   │
├─────────────────────────────────────────────────────────────┤
│  智能体编排层 (LLM Orchestration)                             │
│  意图理解 · 软约束/参数翻译 · 工具调用 · 解释生成 · 不可行诊断 │
├───────────────┬──────────────────────┬──────────────────────┤
│ 预测引擎       │ 决策引擎 (MILP Core)  │ 反馈学习              │
│ Forecasting   │ Optimization          │ Backtest & Calibrate  │
│ 分层直接法     │ 多期现金调度求解       │ 偏差回流 · 采纳回流    │
│ 中性/悲观情景  │ 多方案生成             │ MAPE 看板             │
├───────────────┴──────────────────────┴──────────────────────┤
│  领域服务层 (Domain Services)                                 │
│  头寸/账户 · 现金流项 · 品种主数据 · 授信额度 · 备付规则       │
├─────────────────────────────────────────────────────────────┤
│  数据集成层 (Integration)                                     │
│  标准数据契约 · 适配器(ERP/银企/市场) · 数据质量门 · 落库      │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 LLM 与优化器的职责边界

**关键设计原则：LLM 不做数值最优化，优化器不做语言理解。**

- **LLM（编排者/翻译官）**：
  1. 意图理解：把自然语言诉求映射到“查询 / 预测 / 求解 / 解释”动作；
  2. 参数与软约束翻译：把“先别动那笔并购款”转成对 `t=k` 的资金锁定约束；
  3. 工具调用：触发预测引擎、构造优化模型、调用求解器；
  4. 解释生成：把最优解 + 影子价格翻译成业务叙事与推理链；
  5. 不可行诊断：把求解器的 infeasibility 翻译成可读的“缺口位置/规模/建议”。
- **优化器（决策大脑）**：在硬约束下求全局最优，并返回最优解、目标值、对偶/影子价格、敏感性。

> ⚠️ 避免反模式：不要让 LLM “拍脑袋”给资金配置数字。所有金额级决策必须来自优化器，LLM 只负责建模输入与解释输出。

## 2. 核心数据模型（Canonical Model）

设计原则：源系统异构（SAP/Oracle/用友/金蝶/各行直连），**核心引擎只认下述标准模型**，适配层负责映射。所有实体预留 `entity_id` 以备多法人扩展。

### 2.1 实体关系（概念）

```
Entity (法人主体, MVP=1)
  ├── Account (账户) ──< BalanceSnapshot (余额快照)
  ├── CashFlowItem (现金流项) ── 预测的输入与输出
  ├── Instrument (投融资品种主数据)
  ├── CreditLine (授信额度)
  ├── ReserveRule (备付金规则)
  ├── Forecast (预测结果, 分层+情景)
  ├── StrategyPlan (策略方案) ──< PlanAction (方案动作)
  └── AuditLog (审计留痕)
```

### 2.2 关键表/对象字段

**CashFlowItem（现金流项 —— 预测的原子）**

| 字段 | 类型 | 说明 |
|---|---|---|
| `item_id` | string | 唯一标识 |
| `entity_id` | string | 法人维度（扩展点） |
| `direction` | enum(`inflow`/`outflow`) | 收 / 付 |
| `category` | enum | 销售回款/采购付款/薪酬/税费/利息/还本/租金/其他 |
| `source_type` | enum(`contract`/`ar`/`ap`/`order`/`schedule`/`statistical`) | 来源，决定所属预测层 |
| `expected_date` | date | 预期发生日（映射到周次 t） |
| `amount` | decimal | 金额 |
| `certainty_layer` | enum(`deterministic`/`pattern`/`uncertain`) | 第 1–4 / 5–8 / 9–13 周 |
| `currency` | string | 币种（MVP 本币为主） |

**Instrument（品种主数据 —— 策略动作的原子）**

| 字段 | 类型 | 说明 |
|---|---|---|
| `instrument_id` | string | 唯一标识 |
| `kind` | enum(`invest`/`finance`) | 投资 / 融资 |
| `liquidity_tier` | enum(`cash`/`stable`/`yield`) | 活钱/稳健/增益层（仅投资） |
| `rate` | decimal | 年化收益率（投）/ 成本率（融） |
| `tenor_options` | int[] | 可选期限（周），如 `[1,2,4,8,12]`；`0`=T+0 活期 |
| `min_amount` / `max_amount` | decimal | 起投/起提与上限 |
| `redeemable` | bool | 是否可提前赎回 |
| `redeem_cost` | decimal | 提前赎回成本（如有） |
| `counterparty` | string | 对手方（集中度归类用） |
| `whitelisted` | bool | 是否在白名单 |
| `finance_priority` | int | 融资优先级（成本升序，越小越先用） |

**CreditLine（授信）**：`line_id, entity_id, instrument_ref, limit, used, available, rate`

**ReserveRule（备付金规则）**：`rule_type(fixed/rolling_coverage), value/weeks, entity_id`
> MVP 推荐 `rolling_coverage`：最低备付 = 未来 N 周刚性支出之和。

**Forecast（预测结果）**：按 `week_t × scenario(neutral/pessimistic)` 存净现金流点值/区间，并保留分层明细以支持解释。

**StrategyPlan / PlanAction**：方案头（风险档位、预期净收益、安全垫曲线、预警标签）+ 明细动作（`t, instrument_id, action(invest/redeem/draw/repay), amount, tenor`）。

## 3. 决策引擎：多期现金调度 MILP

### 3.1 问题陈述

在 13 周时间轴上排布投/融/留，使全周期净财务收益最大，且每周满足流动性安全与各项额度约束。**“闲置资金运用”与“缺口弥补”是同一模型的一体两面。**

### 3.2 集合与参数

```
集合
  T = {1..13}                  周次
  I = 投资品种集合              (kind=invest, whitelisted)
  F = 融资工具集合              (kind=finance)
  D_i = 品种 i 的可选期限集合    (单位: 周)

参数
  CF[t]        第 t 周的预测基础净现金流（中性情景，来自预测引擎）
  CF_pess[t]   第 t 周的悲观净现金流（用于安全压测/预警）
  B0           期初可用现金
  r[i]         投资品种 i 的周化收益率
  c[f]         融资工具 f 的周化成本率
  MinCash[t]   第 t 周最低备付金（来自 ReserveRule）
  Limit[f]     融资工具 f 可用额度
  Lock[t]      第 t 周被司库锁定、不可动用的资金
  riskKnob     风险偏好（影响 MinCash 厚度与久期上限）
```

### 3.3 决策变量

```
x[i,t,d] ≥ 0   第 t 周投入品种 i、期限 d 周的金额
y[f,t]   ≥ 0   第 t 周动用融资工具 f 的金额
red[i,t] ≥ 0   第 t 周提前赎回品种 i 的金额（仅 redeemable 品种）
B[t]     ≥ 0   第 t 周末可用现金余额
（可选 z[i,t] ∈ {0,1} 用于最小起投额等整数约束 → 故为 MILP 而非纯 LP）
```

### 3.4 现金守恒（核心约束，∀ t∈T）

```
B[t] = B[t-1] + CF[t]
       + Σ_{i,d : t是到期周} x[i, t-d, d]·(1 + r[i]·d)     // 投资到期本息回流
       + Σ_i red[i,t]·(1 - redeem_cost[i])                 // 提前赎回回流
       - Σ_{i,d} x[i,t,d]                                  // 本周新投出
       + Σ_f y[f,t]                                        // 本周新增融资
       - 融资到期还本付息项
   (B[0] = B0)
```

### 3.5 约束清单

**硬约束（必做）**

```
(C1) 流动性安全:   B[t] ≥ MinCash[t]                         ∀ t
(C2) 授信额度:     Σ_t y[f,t] ≤ Limit[f]                     ∀ f
(C3) 投资白名单:   仅 whitelisted 品种进入 I
(C4) 资金锁定:     B[t] ≥ MinCash[t] + Lock[t]               ∀ t（被锁资金不可动用）
(C5) 起投/上限:    min_amount·z ≤ x ≤ max_amount·z          （触发整数变量 z）
(C6) 可赎回限制:   red[i,t] ≤ 截至 t 仍在存续的品种 i 持仓
```

**软约束 / 后置（精修）**

```
(C7) 集中度:       Σ 单对手方持仓 ≤ ρ · 总投资
(C8) 流动性分层:   Σ_{i∈cash} 持仓 ≥ λ · 总投资
(C9) 期限错配惩罚: 目标中对长久期占用加惩罚项
```

### 3.6 目标函数

```
max  Σ_{i,t,d} x[i,t,d]·r[i]·d            // 投资利息收益
   − Σ_{f,t}   y[f,t]·c[f]·(占用周数)      // 融资成本
   − Σ_i red[i,t]·redeem_cost[i]          // 赎回成本
   [ − 期限错配惩罚 (C9, 可选) ]
```

`riskKnob` 的作用：
- **进取**：调低 `MinCash[t]`、放宽久期上限 → 贴近安全线榨收益；
- **稳健**：调高 `MinCash[t]`、压短久期 → 加厚安全垫。

### 3.7 缺口预警（软信号，不影响可行性）

求解后，对**悲观情景**复算每周安全垫与融资依赖度：

```
safety_cushion_pess[t] = B_pess[t] − MinCash[t]
若 safety_cushion_pess[t] < warn_threshold  → 打“缺口预警”标签
若 Σ y[f,t] / 总可用授信 > dep_threshold     → 打“高融资依赖”标签
```

### 3.8 多方案与不可行处理

- **多方案**：以不同 `riskKnob`（稳健/折中/进取）多次求解 → 2–3 套对比方案。
- **不可行诊断**：当 (C1) 在用满 (C2) 后仍无法满足时，模型不可行。此时返回**最小缺口松弛**（引入松弛变量 `slack[t]` 并最小化 `Σ slack[t]`），定位缺口的周次与规模，交 LLM 翻译为可读建议。

### 3.9 求解器选型建议

- 模型规模小（13 周 × 数十品种），CBC / HiGHS（开源）即可秒级求解；商用 Gurobi 为可选增强。
- 建模层用 PuLP / OR-Tools / Pyomo 任一，封装为 `solve(model_input) -> solution` 服务。
- 提供对偶值/影子价格输出（如 C1、C2 的影子价），支撑“授信再多 X 能多赚 Y”的敏感性解释。

## 4. 预测引擎设计

### 4.1 分层方法路由

| 层 | 周次 | 数据来源 | 方法 |
|---|---|---|---|
| 确定层 | 1–4 | AR/AP 到期、合同、收付日历 | 规则/日程驱动逐笔汇总 |
| 规律层 | 5–8 | 历史回款账龄、采购节奏 | 账龄转化模型 + 时序统计（季节/趋势） |
| 不确定层 | 9–13 | 订单/销售预测 + 历史 | 业务驱动 + 统计模型，**输出区间** |

### 4.2 双情景输出

- **中性**：各层期望值汇总，作为优化求解的输入 `CF[t]`。
- **悲观**：对不确定性较高的收入项打折/延后、对支出项从紧，生成 `CF_pess[t]`，用于安全压测与缺口预警。
- 数据结构：`Forecast[week_t][scenario] = {point, lower, upper, layer_breakdown}`。

### 4.3 反馈纠偏闭环

```
每周滚动:
  1. 锁定上周已“实际发生”的现金流, 计算 预测误差 = 实际 − 预测
  2. 更新 MAPE 看板 (分层、分收支类)
  3. 将系统性偏差回流模型 (如对某客户回款速度的高估系数)
  4. 用最新数据重算未来 13 周, 触发决策引擎重优化
```

### 4.4 预测引擎接口（内部）

```
forecast(entity_id, as_of_week, horizon=13)
  -> { weeks: [ {t, neutral:{...}, pessimistic:{...}, items:[...] } ],
       accuracy: { mape_by_layer, mape_by_category } }
```

## 5. LLM 编排层设计

### 5.1 动作目录（Tool/Function）

| 工具 | 入参 | 出参 |
|---|---|---|
| `run_forecast` | entity, as_of, horizon | 分层+双情景预测 |
| `build_and_solve` | forecast, riskKnob, locks, overrides | StrategyPlan(最优解+对偶值) |
| `explain_plan` | StrategyPlan | 推理链文本（业务叙事） |
| `diagnose_infeasible` | 松弛解 | 缺口位置/规模/建议文本 |
| `query_position` | entity, week | 头寸/安全垫 |
| `apply_overrides` | 自然语言→约束/参数 | 修改后的模型输入（what-if） |

### 5.2 典型对话流（示例）

```
用户: “下个月底那笔并购款付得出来吗? 不够的话怎么补最省?”
 → LLM 识别: 时点约束(锁定并购款) + 缺口诊断 + 弥补策略
 → apply_overrides(锁定 t=并购周 的资金)
 → run_forecast → build_and_solve
 → 若可行: explain_plan (给弥补方案 + 成本对比)
 → 若不可行: diagnose_infeasible (报缺口规模与建议)
```

### 5.3 约束防线（安全）

- LLM 不得绕过硬约束输出方案；所有金额决策来自 `build_and_solve`。
- 用户/外部内容中的“指令”不自动执行；涉及参数变更需在交互层确认。
- 全程动作写入 `AuditLog`。

## 6. 数据集成层

### 6.1 数据需求与来源

| 消费方 | 所需数据 | 来源 |
|---|---|---|
| 预测引擎 | AR/AP+账龄、订单、合同、收付日历、历史流水 | ERP/财务系统、银企直连 |
| 头寸/流动性 | 实时/日终余额、账户、受限资金 | 银企直连/网银 |
| 决策引擎 | 品种主数据、授信额度、市场报价 | TMS/手工维护/市场数据源 |
| 反馈学习 | 历史预测 vs 实际 | 系统内落库 |

### 6.2 适配器模式

```
SourceAdapter (SAP / Oracle / 用友 / 金蝶 / 各行直连)
        │  映射
        ▼
Canonical Model (CashFlowItem / Account / Instrument / CreditLine ...)
        │
        ▼
Data Quality Gate (完整性/异常值/口径校验)  ──失败──▶ 拦截 + 告警
        │  通过
        ▼
Domain Store (落库, 供引擎消费)
```

### 6.3 数据质量门规则（示例）

- 完整性：必需收支项是否缺失（如本周应付为空需复核）；
- 异常值：金额数量级离群检测；
- 口径一致：币种、科目映射、日期→周次归集正确；
- 时效：余额（实时/日终）、单据（T+1 可接受）、市场报价（按日）。

### 6.4 写回策略

参谋不执行交易；写回限于**记录性回写**（预测、方案存档），不产生任何交易指令。

## 7. 非功能性需求（技术指标）

| 类别 | 指标/要求 |
|---|---|
| 准确性 | 分层 MAPE 阈值（近端最严、远端考核区间命中率）；MILP 返回精确最优 |
| 性能 | 单体 13 周模型秒级求解；预测+求解端到端 < 数秒；对话响应低延迟 |
| 可解释性 | 每条建议附 数据→约束→目标→解 的可追溯链；提供影子价格解释 |
| 可审计性 | `AuditLog` 记录问答、建议、采纳/否决，不可篡改、可检索 |
| 安全权限 | 财务数据加密；RBAC 控制 查看/改假设/调风险偏好；访问留痕 |
| 可靠性 | 数据源中断降级（用上次数据+陈旧标记）；无解给可读诊断而非报错 |
| 可观测性 | 预测准确度看板、求解耗时、采纳率埋点 |

## 8. 扩展点设计（面向集团演进）

| 扩展点 | MVP 处理 | 集团升级 |
|---|---|---|
| 法人维度 | 所有实体带 `entity_id`（单值） | 多 `entity_id` + 合并视图聚合 |
| 决策动作 | 动作类别可插拔注册 | 新增“内部调剂”类（上存下借/委贷） |
| 优化模型 | 变量按 `(i,t,d)` 组织 | 变量加实体维度 `(e,i,t,d)`，增内部转移变量 |
| 约束 | 约束以规则对象注册 | 增资金池归集、内部额度、跨境/外管约束 |
| 视图 | 单主体视图 | 双层（集团合并 + 成员明细） |

> **升级原则**：从单体到集团主要是“加维度、插动作、增约束”，引擎结构不变。

## 9. 模块清单与交付边界（MVP）

| 模块 | 关键交付 | 优先级 |
|---|---|---|
| 数据集成层 | 标准模型 + 至少 1 个 ERP 适配器 + 银企直连 + 质量门 | P0 |
